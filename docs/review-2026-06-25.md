## 1. Executive Summary

Ten days of focused work produced two validated pipeline versions. The DER Refinement Agent (formerly V1.0) matched structured DER Excel rows against flat OH products (1,000-row production run, 54.8% match rate). The Pre-DER Agent (formerly V2.0) shifted to free-form voice inputs matched against the PN hierarchy tree (20-input experiment, 95% match rate, 80% High-confidence top-1). Both agents run end-to-end, are cost-efficient (DeepSeek only, zero failures in all runs), and the core design decisions are well-documented in three ADRs.

The major outstanding risk is not in the code — it is in the **gap between what the system currently tests and what production needs**: structured field exploitation, hardware PN coverage, the D365 writeback / feedback loop, and wiring Pre-DER Agent output into the DER Refinement Agent's recall pool. Human validation of the Pre-DER Agent PPT output is the next critical gate.

## 3. Code Quality — What is Working Well

### 3.1 Architecture
The 3-stage pipeline (BM25 + dense recall → LLM rerank → confidence scoring) is a correct design for this retrieval problem. The separation into `src/` modules with clear responsibilities means V2.0 was able to reuse all four core modules with minimal modification.

### 3.2 Hallucination Mitigation (ADR-0003)
Replacing `product_id` string return with positional `candidate_no` integers was the right call. The LLM cannot fabricate a valid product ID, cannot swap IDs silently, and the prompt is shorter. The parser in `rerank.py:155–175` correctly bounds-checks, deduplicates, and maps positions back to candidate objects.

### 3.3 Threshold Tuning (ADR-0002)
The evidence-based threshold change (0.30 → 0.40) was well-reasoned. The 190-slot floor-clustering at 0.30–0.39 is a real signal that the LLM was using 0.30 as a "I must give something" floor. The ADR documents this clearly for future reference.

### 3.4 Checkpointing in DER Refinement Agent (`run_der_refinement_agent.py`)
Every-20-row checkpointing with atomic `tmp → rename` write is production-grade. The resume logic correctly re-runs failed/skipped rows. This was needed and delivered.

### 3.5 Backward Compatibility
Pre-DER Agent extensions to `recall.py` (`corpus_texts=` kwarg), `rerank.py` (`prompt_path=`, `format_fn=` params), and `confidence.py` (`keep_topk_diverse_tree()`) are all additive. DER Refinement Agent's `run_der_refinement_agent.py` still works unchanged.

### 3.6 Pre-DER Agent Diversity Function
`keep_topk_diverse_tree()` correctly detects ancestor-descendant conflicts using path prefix matching. The deferred-candidate fallback (if <3 diverse picks) is correct and prevents empty slots when the top candidates cluster in one branch.

### 3.7 PPT Generation
`generate_ppt.py` is well-designed: left panel shows raw voice + extracted description, right panel shows top-3 tree nodes with confidence badges and rank colors. This is exactly the right artifact for Helen to review and annotate without reading raw JSON. 80+ lines of PPTX-construction code with no third-party charting library dependency.

## 4. Bugs and Code Issues

### 4.1 `recall.py:55` — `recall()` Return Bound is Broken (Medium) ✅ FIXED 2026-06-25

```python
return union_sorted[: max(topk, len(union_sorted))]
```

`max(topk, len(union_sorted))` equals `len(union_sorted)` whenever the union exceeds `topk`, so the slice does nothing — all candidates are returned. The `topk` parameter does not limit output.

**Why it doesn't break things today**: Both runners slice the result externally:
```python
cand_indices = idx.recall(row.description or "", topk=RECALL_TOPK)[:RERANK_TOPN]
```
The `[:RERANK_TOPN]` cap in the caller ensures ≤30 candidates reach the LLM. But the function signature implies a limit that isn't enforced, which is confusing for future callers.

**Recommendation**: Change to `return union_sorted[:topk]` if the intent is a soft cap, or rename the parameter to make the behavior explicit.

---

### 4.2 `recall.py:54` — Union Ranking Loses Score Information (Medium) ✅ FIXED 2026-06-25

```python
union_sorted = sorted(union, key=lambda i: -(1 if i in ids_dense else 0))
```

This is a binary sort: dense results go first, BM25-only results go second. Within each group, ordering is arbitrary (set iteration order / list order). The actual cosine scores and BM25 scores are discarded.

**Implication**: Candidates 1–30 from dense recall are ranked above candidates 1–30 from BM25 recall, but within the dense group the best dense match could be ranked 30th if it happened to be last in `order`. Same for BM25. The RERANK_TOPN slice may exclude high-quality BM25-only candidates that ranked lower due to arbitrary ordering.

**Recommendation**: Keep the `cos` array in scope after the dense computation and use it for scoring:
```python
union_sorted = sorted(union, key=lambda i: -(cos[i] if cos is not None else 0.0))
```
For BM25-only items, use a normalized BM25 score or place them after all dense items with their BM25 score as tiebreaker.

This script is a one-shot preprocessor that converts `data/raw/sales-oppty-talking-script.md` → `data/converted/sales-voice-inputs.md`. It is not mentioned in `CLAUDE.md`, not in `scripts/`, and uses hardcoded relative paths. It was clearly run once on June 23 and never again.

**Recommendation**: Move to `scripts/`, add a one-line docstring, mention it in CLAUDE.md.

### 4.6 Pre-DER Agent `Candidate` Field Repurposing — Code Smell (Low)

In `run_pre_der_agent.py:67–76`, `node_to_candidate()` maps `PNNode` attributes into DER Refinement Agent's `Candidate` fields with misleading names:

```python
Candidate(
    product_id=str(idx),           # actually: corpus index (int as str)
    product_name=node.name,        # OK
    parent_product=" > ".join(node.path),  # actually: full ancestor path
    solution_category=f"L{node.level}",   # actually: tree level label
    solution_sub_category=sample_pns,     # actually: sample PN descriptions
    iso=None,
)
```

`format_candidates_block_v2` then interprets these repurposed fields in `rerank_v2.txt`. This works correctly — the roundtrip is consistent — but any future maintainer reading `Candidate.solution_category` will be confused by a value like "L3".

**Recommendation**: Either add a Pre-DER-specific dataclass (`TreeCandidate`) in a future refactor, or add inline comments at the `node_to_candidate()` definition explaining the field mapping.

### 4.7 `load_data.py` — Brittle Hard-Coded Column Numbers (Technical Debt)

All column reads in `load_oh()` and `load_der()` use literal integer indices:
```python
status = _clean(ws.cell(row=r, column=16).value)
bg = _clean(ws.cell(row=r, column=18).value)
```

If someone adds or reorders a column in the Excel file, these break silently (return wrong data with no error). This has already surfaced in the meeting transcript: the DER header was truncated and a column (`is_opportunity_an_extension_of_existing_true_scale`) was missing from the data provided.

**Recommendation**: Read the header row once and build a column-name-to-index dict. Use named lookups: `col["Business Group"]` instead of `column=18`.

## 6. Design Gaps vs. Production Requirements

### 6.1 Structured Field Exploitation (Critical Gap)

The Helen & Ziff transcript (June 25) documents a multi-field decision cascade that is **not implemented**:

```
Priority order:
  Col 4  (BG) → hard partition
  Col 22 (Service Model) → main product category
  Col 16 (Existing true-scale contract?) → narrows to DaaS/ISS
  Col 10 (AI flag) → adds AI products
  Col 12 (Asset recovery flag) → adds ARS products
  Col 5  (Business problem text) → last-resort soft signal
```

The current DER Refinement Agent pipeline uses **only Col 5** (business problem text). The Pre-DER Agent pipeline uses Col 5 after LLM extraction, plus the explicit BG field from the voice input file. 

This is the single largest precision improvement available without changing the model or the PN tree. A rule-based pre-filter using Cols 22/16/10/12 before the recall stage would eliminate a large fraction of mismatches. Helen's domain knowledge about these rules is now captured in the transcript but is not encoded anywhere in the codebase.

**Recommendation**: Create `docs/field-logic.md` that formalizes Helen's cascade as pseudocode. This is a prerequisite for the next sprint.

### 6.2 Hardware PN Coverage (Known Gap)

The current OH product list (`OH product in D365.xlsx`) and the PN tree (`Advanced PN List.xlsx`) are primarily **services**. Hardware PNs are mentioned in the transcript:

> "現在我們還缺少硬件的" (we're still missing hardware)

This is a known gap. The IDG match rate in the DER Refinement Agent (63.4%) vs DCG (46.2%) may partly reflect that hardware-heavy IDG deals have no matching OH products in the services-focused catalog.

### 6.3 D365 Writeback and Feedback Loop (Open Issue from Meeting)

From `meeting-notes-agreed-actions.md`, item #9 is still Open:
> A reviewable field writeback rule must be established between D365 and the analytics data table, so semantic matching never silently drops key columns.

Without a writeback mechanism, human reviewers manually re-enter the AI recommendation into D365. Without a feedback signal (did the seller accept or reject the #1 recommendation?), the system cannot improve over time. The transcript explicitly calls this out:

> "要从真正的使用者反馈出来的，那个数据要重新去调整我们的方案"  
> (The scheme must be adjusted based on real user feedback)

### 6.4 Pre-DER Agent Has No Checkpoint/Resume

The DER Refinement Agent checkpoints every 20 rows. The Pre-DER Agent has no checkpoint logic. For 20 inputs this is fine. For the next phase (more inputs, or a real-time agent loop), a mid-run failure loses all progress.
