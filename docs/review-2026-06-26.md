## 4. New Code — `src/field_rules.py`

### 4.1 Correctness

The implementation is correct and matches the design in `docs/field-logic.md`:

- **ARS guarantee** (`is_ars is True` OR scope contains ARS string): correct. `_yn_to_bool()` in `load_data.py` converts "Yes"/"No" strings to `bool`, so `is True` works as intended.
- **DaaS guarantee** (`service_model == "DAAS"`): correct, exact match after `.strip().upper()`.
- **Boost logic** (AI, existing expansion, scope): all correctly wired to keyword lists.
- **`boosted_ids -= guaranteed_ids`**: correctly prevents double-listing.
- **`inject_field_candidates` merge order**: Guaranteed → Boosted → Recall remainder, trimmed to `max_candidates=30`. Order is correct.

### 4.2 Minor Code Issues

**Double BG filter (Low, cosmetic)**

`apply_field_rules(row, idx.products)` passes `bg=row.business_group` into `_find_products()`, but `idx.products` is already BG-partitioned (each `RecallIndex` holds products for one BG only). The BG filter in `_find_products` re-filters an already-filtered list — harmless, but redundant. It only causes a real difference if `row.business_group` and `p.business_group` differ in whitespace or casing, which `_clean()` makes unlikely.

**`pid_to_idx` silent overwrite (Low)**

In `inject_field_candidates`:
```python
pid_to_idx = {p.product_id: i for i, p in enumerate(products)}
```
If two OH products share the same `product_id` (a data quality issue rather than a code bug), the last one wins silently. The OH Excel has 2,049 active rows; a spot-check is warranted but this is low risk.

**`_scope_matches` substring test (Low, accepted)**

`key in scope.upper()` uses substring matching. The four known scope strings have no overlapping substrings, so false positives are not possible in the current dataset. If a new scope value is added that contains a substring of an existing key (e.g., one containing "ASSET RECOVERY"), it could trigger incorrectly. Noted in `field-logic.md §7` — acceptable as a known limitation.

**`_SERVICE_MODEL_KW` missing SAAS (Low)**

Service Model value `"SAAS"` appears in the 2026 DER data (≈1 occurrence per `field-logic.md §6`). It has no entry in `_SERVICE_MODEL_KW`, so SAAS deals receive no guaranteed or boosted candidates — they fall through to pure text recall. Current behavior is correct (no crash, no incorrect injection), but the gap should be documented as a TODO until the correct OH product mapping for SAAS is identified.

### 4.3 Integration in `run.py`

The integration is clean:

```python
raw_recall = idx.recall(row.description or "", topk=RECALL_TOPK)   # 60 candidates
rules = apply_field_rules(row, idx.products)
cand_indices = inject_field_candidates(raw_recall, idx.products, rules, max_candidates=RERANK_TOPN)  # → 30
```

- `RECALL_TOPK` was raised from 30 → 60 to give field injection more candidates to merge before trimming to `RERANK_TOPN=30`. This is the correct approach — field-guaranteed products may not appear in the top-30 recall results, so the recall pool needs to be wider.
- `field_guaranteed` and `field_boosted` counts are stored in the output JSON per row, enabling future analysis of injection frequency.
- The field rules run after recall and before the LLM — correct position in the pipeline.

## 7. Test Run Analysis — `test_fixes` (20 rows, 2026-06-25 12:19)

### 7.1 Match Rate vs Baseline

| Metric | `test_fixes` (20 rows) | `1000` baseline (1000 rows) | Change |
|---|---|---|---|
| Match rate | 65.0% | 54.8% | **+10.2 pp** |
| Avg kept score | 0.879 | 0.710 | **+0.169** |
| Top-1 High | 100% of matched | 69.9% of matched | Higher |
| All-3-unique-parent | 4/13 (31%) | 365/548 (67%) | **Regression** |
| All-3-same-parent | 7/13 (54%) | ? (baseline not reported) | **High** |

The 10 pp match-rate gain is encouraging, but the 20-row sample is too small for statistical significance. The much higher average score (0.879 vs 0.710) suggests field injection is correctly surfacing highly relevant candidates that previously ranked outside the top-30 recall window.

### 7.2 Diversity Regression

The same-parent rate of 54% (7 of 13 matched rows returning all 3 candidates from the same parent) is a significant regression from the 1000-row baseline behavior. Root cause:

Field injection places guaranteed/boosted products at the **head** of the candidate list. If an ARS or DaaS guarantee injects 2–3 products that share a parent (e.g., multiple DaaS variants under "Device as a Service"), the LLM sees them first and scores them High. The post-rerank `keep_topk_diverse()` then finds the top-3 slots already taken by same-parent products and cannot undo the clustering.

The diversity function operates on the rerank output, not on the injection input. It works by iterating through ranked candidates and skipping those whose parent is already represented. But if all candidates above the threshold share one parent, there are no diverse alternatives to substitute.

**Recommendation**: Apply a pre-injection diversity constraint or expand `inject_field_candidates` to interleave guaranteed candidates rather than stacking them. Alternatively, increase `RERANK_TOPN` for rows with many guaranteed candidates so the LLM sees enough diverse alternatives to fill 3 distinct-parent slots.

### 7.3 Bimodal Confidence Distribution

All 34 kept slots are High (32) or Medium (2). Zero Low. This is a stark contrast to the baseline where Medium and Low slots were more common. The bimodal pattern (High or None) suggests field injection is creating a step function: rows with a strong service-model signal get very confident matches; rows without one get nothing at all. This may be correct behavior, or it may indicate the diversity regression is masking medium-confidence candidates that should appear.

## Addendum (2026-06-29) - Copilot Studio DER Issue Closure

### What was closed

- The DER Topic issue where `CountRows(Topic.DerApiResponse.topk)=0` for variable-based requests is now closed.

### Root cause

- In Copilot Studio DER requests, optional fields can be serialized as JSON `null` when left empty.
- For DER payloads, `scope` (and similarly optional `service_model`) must follow empty-string semantics; incoming `null` triggered request-model validation failure (HTTP 422), which surfaced as empty `topk`.

### Fix applied

- Backend request model in `app.py` now defensively normalizes `null -> ""` for `scope/service_model` before downstream matching.

- `data/raw/copilot-studio-session-summary.md`: updated from unresolved status to resolved, with root cause and fix details.
